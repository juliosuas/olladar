## Install

Requirements:

| Component | Minimum version | Why |
|---|---|---|
| Node | 22.5 | `node:sqlite` is built-in, no npm install needed |
| ollama | 0.5+ | stable `/api/chat` + `/api/generate` with `think` option |
| Python | 3.10 | CLI helpers; eval harness uses stdlib + PyYAML |
| Linux with systemd user | any modern | only needed for autostart; the proxy runs fine standalone |

Clone and run the installer:

```bash
git clone https://github.com/juliosuas/olladar.git
cd olladar
./install.sh
```

Run manually without systemd:

```bash
node --experimental-sqlite bin/olladar-proxy.mjs
```

## Configuration

olladar reads no config file. Everything it does is controlled by constants at the top of `bin/olladar-proxy.mjs`:

```js
const UPSTREAM = { host: '127.0.0.1', port: 11434 };  // ollama
const LISTEN_PORT = 11435;                            // where olladar listens
```

If you need to change these, edit the file and restart the service:

```bash
systemctl --user restart olladar-proxy.service
```

### Disabling `think: false` injection

Set `think: true` explicitly in your client's request body. olladar only injects when `think` is unset AND `tools[]` is present.

### Running against a remote ollama

Change `UPSTREAM` to point at the remote host. Note: olladar does not add TLS or auth — run it on a trusted network.

## CLI reference

The `olladar` command queries the SQLite log. Output is plain-text and pipe-friendly.

| Subcommand | Usage | What it shows |
|---|---|---|
| `last [N]` | `olladar last 20` | The N most recent calls with TTFT, totals, tokens, status |
| `slow [N]` | `olladar slow 10` | The N slowest calls — finds regressions and outliers |
| `failures [N]` | `olladar failures 30` | Calls with errors, 4xx/5xx, or tools requested but not emitted |
| `stats [--since DUR]` | `olladar stats --since 24h` | Aggregate per model: p50/p95 latency, avg tokens, tool-use rate |
| `tools [--since DUR]` | `olladar tools --since 7d` | Per-model tool-use accuracy (emission rate) |
| `trace <id>` | `olladar trace a1b2c3d4...` | Full messages, tools array, and response for one request |
| `size` | `olladar size` | DB file size and row counts |
| `watch` | `olladar watch` | Live tail (polls every 1s) of new calls as they complete |
| `stream "<prompt>"` | `olladar stream --think off "hola"` | Send a prompt directly with live tok/s counter |

Duration format: `Nh`, `Nd`, `Nm` (e.g. `2h`, `7d`, `30m`). Defaults to 24h where optional.

### `olladar stream` flags

| Flag | Default | Effect |
|---|---|---|
| `--model NAME` | `qwen3:32b` | Any model ollama has loaded or can load |
| `--endpoint URL` | `http://127.0.0.1:11435` | Any ollama-compat endpoint |
| `--think on\|off\|default` | `default` | Override the `think` flag |
| `--max-tokens N` | 1000 | `num_predict` cap |
| `--temp F` | 0.4 | Temperature |
| `--keep-alive DUR` | `24h` | How long ollama should keep the model loaded after the call |
| `--tools` | off | Add a dummy tool to trigger olladar's auto `think:false` path |

## SQLite schema

`~/.local/share/olladar/logs.sqlite` contains two tables.

### `calls`

One row per HTTP request, created on request arrival and finalized on response end.

| Column | Type | Notes |
|---|---|---|
| `request_id` | TEXT PK | UUID generated by olladar |
| `ts` | INTEGER | Epoch ms of request arrival |
| `endpoint` | TEXT | e.g. `/api/chat`, `/api/generate` |
| `model_requested` | TEXT | What the client sent in `body.model` |
| `model_served` | TEXT | What ollama reported in its response |
| `had_tools` | INTEGER | 1 if `body.tools[]` was a non-empty array |
| `tool_call_emitted` | INTEGER | 1 if the response contained structured `tool_calls` |
| `stream` | INTEGER | 1 if `body.stream === true` |
| `ttft_ms` | REAL | Time from request arrival to first byte of the response |
| `total_ms` | REAL | End-to-end wall-clock time |
| `prompt_tokens` | INTEGER | From `prompt_eval_count` / `usage.prompt_tokens` |
| `completion_tokens` | INTEGER | From `eval_count` / `usage.completion_tokens` |
| `fallback_used` | INTEGER | Reserved for a future hybrid-routing feature |
| `status_code` | INTEGER | HTTP status from upstream |
| `error` | TEXT | Short error string on failures |

### `traces`

> ⚠️ **The `traces` table contains the full text of every prompt sent through olladar,** including anything users paste (API keys, credentials, PII, internal docs). On a shared machine, encrypted backup, or repurposed disk, this is a real disclosure surface. If this is a concern:
>
> - **Purge logs older than 7 days:** `sqlite3 ~/.local/share/olladar/logs.sqlite "DELETE FROM traces WHERE ts < strftime('%s','now')*1000 - 7*86400*1000"`
> - **Disable trace storage entirely:** edit `~/.local/bin/olladar-proxy.mjs` and comment out the `insertTrace.run(...)` block. Keeps `calls` (metrics only) intact.
> - **Delete everything:** `./uninstall.sh --purge`

Full bodies for analysis and potential fine-tuning dataset export. Skipped when `messages` serializes to > 500 KB (a sentinel row records the skip reason).

| Column | Type | Notes |
|---|---|---|
| `request_id` | TEXT PK | FK to `calls.request_id` |
| `ts` | INTEGER | Mirror of `calls.ts` |
| `messages_json` | TEXT | `body.messages` as JSON |
| `tools_json` | TEXT | `body.tools` as JSON |
| `response_json` | TEXT | Final JSON object from upstream (last line of NDJSON for streams) |

Query directly with any SQLite tool. Example — all tool-use failures in the last 7 days:

```sql
SELECT c.ts, c.model_served, t.messages_json
FROM calls c JOIN traces t USING(request_id)
WHERE c.had_tools = 1 AND c.tool_call_emitted = 0
  AND c.ts > strftime('%s','now')*1000 - 7*86400*1000
ORDER BY c.ts DESC;
```

## The `think: false` injection rule

Located in `bin/olladar-proxy.mjs`:

```js
if (hadTools && bodyObj && bodyObj.think === undefined) {
  bodyObj.think = false;
  thinkInjected = true;
  forwardBodyBuf = Buffer.from(JSON.stringify(bodyObj));
}
```

**Why it matters.** qwen3 models with default `reasoning=true` fill their `num_predict` budget with `thinking` tokens when `tools[]` is present. They never get to emit `tool_calls`. Measured on RTX 4090 at ctx 4K:

- Tool-use accuracy: 38 % (3/8) → **100 %** (8/8) with `think: false`
- Average wall time for tool decisions: 7.2 s → **1.3 s**
- Overall olladar eval harness pass rate: 35 % → **70 %**

**When you want reasoning.** For synthesis, code generation, and complex questions, `think: true` (the default) is correct — reasoning improves quality. The rule fires only when the request is clearly asking the model to call a tool. Tool schemas are short-form instructions; reasoning there is overhead, not quality.

**Opting out.** If you want thinking even for tool requests, set `think: true` explicitly. olladar only injects when `think` is `undefined`.

## Eval harness

`eval/run.py` reads `eval/eval-cases.yaml` and runs every case against a single model, scoring by category-specific heuristics.

```bash
python3 eval/run.py --model qwen3:32b
python3 eval/run.py --model qwen2.5-coder:7b --endpoint http://127.0.0.1:11434
```

Results are saved to `~/Desktop/llm-eval/results/YYYY-MM-DDTHHMM-MODEL.json` and printed to stdout.

### Adding cases

`eval-cases.yaml` is a flat list. Each case has:

```yaml
- id: unique-id
  kind: tool_use | synthesis | coding | long_ctx | whatsapp
  prompt: "user message"
  tools: [browser_open, shell_run, answer]     # optional, for tool_use/whatsapp
  expected_tool: browser_open                  # optional, single or list
  expected_args_contains: ["github"]           # optional, substring match in args
  expected_contains: ["foo", "bar"]            # for synthesis/coding/long_ctx
```

Scoring (see `run.py::score_case`):

- **tool_use / whatsapp with `expected_tool`**: pass if the response has a structured `tool_call` with the expected function name AND args contain the expected substrings. Falls back to JSON-in-content regex match.
- **synthesis / coding / long_ctx**: pass if `max(1, len(expected) // 2)` of the expected substrings appear in the content.

Thresholds are intentionally lenient — the eval is designed to catch regressions, not validate prose quality.

## Integrating your client

Any client that speaks ollama's HTTP API works. Point it at `http://127.0.0.1:11435` instead of `11434`.

### OpenClaw

```json
// ~/.openclaw/openclaw.json
"models": {
  "providers": {
    "ollama-local": {
      "baseUrl": "http://localhost:11435/v1",
      ...
    }
  }
}
```

Then `systemctl --user restart openclaw-gateway.service`.

### Open WebUI

Settings → Connections → Ollama base URL: `http://127.0.0.1:11435`.

### Direct curl

```bash
curl -s http://127.0.0.1:11435/api/chat \
  -H 'Content-Type: application/json' \
  -d '{"model":"qwen3:32b","messages":[{"role":"user","content":"hola"}],"stream":false}'
```

## Troubleshooting

**`SqliteError: unable to open database file`** — `~/.local/share/olladar/` isn't writable. Run `mkdir -p ~/.local/share/olladar && chmod u+w ~/.local/share/olladar`.

**Node throws `--experimental-sqlite is not supported`** — Node < 22.5. Upgrade via nvm: `nvm install 22 && nvm use 22`.

**`status=502 error=ECONNREFUSED`** in logs — ollama isn't running. Start it: `sudo systemctl start ollama`.

**Tool calls still fail 100 % of the time** — the injection only fires when `tools[]` is set by the client. Some frameworks send tools via a different field. Check:

```bash
olladar last 5
# If "had_tools" is 0 and you expect it to be 1, your client isn't sending
# tools in the request body — it's probably using function-calling via system
# prompt or a different schema.
```

**Latency went up after installing olladar** — the logging is synchronous. Expected overhead is < 5 ms per call. If you see more, file an issue with `olladar trace <request_id>` output.

## Extending olladar

The proxy is one file (`bin/olladar-proxy.mjs`) and ~200 lines. Likely extension points:

- **Request-shape routing** — inspect `bodyObj.messages` to decide which model to forward to. Natural home: before the `http.request({...})` call.
- **Response rewriting** — currently pass-through. If you want to parse JSON-in-content into structured `tool_calls`, do it in `proxyRes.on('end', ...)` before calling `res.end()`.
- **Alerting** — long-tail latency alerts could be triggered from the same `on('end')` handler by reading the just-inserted row.

The CLI (`bin/olladar`) is a bash wrapper around Python heredocs. Adding a subcommand means copying an existing case (`last`, `slow`, etc.) and adjusting the SQL.

## FAQ

**Is this an ollama fork?** No. It's a layer on top. It does not modify ollama's code, weights, or config.

**Does it work with other LLM servers?** Anything that speaks ollama's HTTP JSON shape works. vLLM, llama.cpp server in OpenAI-compat mode, etc. all work unchanged — but the `think: false` rule is specific to ollama's option naming.

**Why not LiteLLM?** olladar is single-upstream, zero-rewrite. LiteLLM is a much bigger surface (multi-provider, retries, fallbacks, cost accounting). Use LiteLLM when you need that; use olladar when you don't.

**Why SQLite?** Zero setup, sqlite3 is everywhere, rows are small, 10 MB of logs holds ≈ 20 K calls. If you need more, export to Postgres via `sqlite3 logs.sqlite .dump`.

**Is the `traces` table a fine-tuning dataset?** It can be. Each row has `(messages, tools, response)` — exactly the shape most fine-tuning frameworks expect. Filter where `tool_call_emitted = 1` to get positive examples.
